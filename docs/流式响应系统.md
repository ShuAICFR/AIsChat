# AIsChat 流式响应系统

> **核心原则：用户看到的是"状态"，不是"过程"。**

---

## 设计目标

让 AI 的回复过程有"生命感"——人类打字有停顿、需要组织语言。流式不是炫技，是让用户感知到 AI "正在思考"和"正在打字"的节奏。

传统 AI 聊天是一问一答、一次性吐出全部内容。AIsChat 的 AI 不同：

| 传统方式 | AIsChat 方式 |
|---------|------------|
| 用户发消息 → 等待 → AI 一整段回复突然出现 | 用户发消息 → "正在思考…" → "正在输入中…" → 消息出现 |
| AI 只能回一条 | AI 可以连发多条（想到什么说什么） |
| 看不到 AI 的内部状态 | 前端实时显示 AI 在做什么 |

---

## 架构概览

```
用户发消息
    ↓
WebSocket → 消息队列 → ai_response_worker
    ↓
broadcast: ai_thinking (trigger: "user")
    ↓                             前端显示: "正在思考…"
_tool_call_loop (stream=True)
    ↓
LLM 流式返回 (SSE 逐行解析)
    ├─ reasoning_content → 仅日志记录，不推送
    ├─ content → 累加，不推送（消息最终整段发送）
    └─ tool_calls → 提取后执行
        ├─ send_message/send_dm → broadcast: ai_typing → 前端: "正在输入中…"
        │                           ↓ dispatch → message_sent → 前端: 消息出现
        └─ 其他工具 → 执行，前端继续显示"正在思考…"
    ↓
broadcast: ai_thinking_end → 前端: 清除所有状态
```

### 关键设计决策

| 决策 | 理由 |
|------|------|
| 推理文本不展示 | reasoning 是 AI 的内部思考过程，对用户无意义且干扰阅读 |
| 消息整段发送而非逐字 | send_message 是工具调用——工具执行是原子的，执行完消息出现 |
| 仅用户触发时显示状态 | 闹钟/心跳是 AI 自己的事，用户不需要在对话界面看到 |
| 流式解析用于加速 | 不等完整响应就提取 tool_calls，加速工具调用启动 |
| 上下文压缩（v1.1.0） | 旧消息稳定摘要化，保 prompt cache 命中率 |

## 后端实现

### 1. SSE 流式解析 (`llm_service.py:_chat_completion_streaming`)

使用 `httpx.AsyncClient.stream()` 发起 POST，逐行解析 SSE（`data: {...}\n\n`）。

```
DeepSeek API SSE 格式：

data: {"id":"...","choices":[{"delta":{"reasoning_content":"用户在问..."}}]}
data: {"id":"...","choices":[{"delta":{"content":"我"}}]}
data: {"id":"...","choices":[{"delta":{"content":"觉得"}}]}
data: {"id":"...","choices":[{"delta":{"tool_calls":[{"index":0,"id":"call_1","function":{"name":"send_message","arguments":"{\"content\":\"你好\"}"}}]}}]}
data: {"id":"...","choices":[{"finish_reason":"stop"}],"usage":{...}}
data: [DONE]
```

流式模式下 tool_calls 分多个 chunk 到达，需按 index 累加：
- 首个 chunk：`delta.tool_calls[0].id` + `function.name` + `function.arguments`（首段）
- 后续 chunk：仅 `function.arguments` 增量追加

最终返回与非流式一致的 dict：
```python
{
    "content": "我觉得这是个好主意...",
    "reasoning_content": "用户在问...",  # 仅日志记录
    "tool_calls": [{"id": "call_1", "type": "function", "function": {...}}],
    "usage": {"prompt_tokens": 500, "completion_tokens": 200, ...},
    "finish_reason": "stop"
}
```

### 2. 工具格式校验 (`tool_registry.py:validate_tool_call`)

纯函数，无 DB 访问、无全局状态依赖，可被并发安全调用。

校验顺序：
1. **工具存在性** — 不在注册表中 → 返回错误 + 可用工具列表
2. **必填字段** — 缺少 required 字段 → 返回期望格式 vs 实际收到
3. **字段类型** — 类型不匹配 → 返回字段名 + 期望类型 + 实际类型

不通过时，错误信息注入 tool_calls 循环的下一轮，AI 可据此自我修正。示例：
```json
{
  "error": true,
  "message": "工具 send_message 的参数格式错误：缺少必填字段 content。期望格式：{group_id (integer), content (string)}，实际收到：{group_id: integer}"
}
```

### 3. WebSocket 事件协议

| 事件 | 触发时机 | 字段 | 前端行为 |
|------|---------|------|---------|
| `ai_thinking` | LLM 调用开始 | agent_id, agent_name, group_id/session_id, **trigger** | trigger="user" → 显示"思考中…"气泡 |
| `ai_typing` | 调用 send_message/send_dm 前 | agent_id, agent_name, group_id/session_id, **trigger** | trigger="user" → 切换为"输入中…"气泡 |
| `ai_thinking_end` | 工具调用循环结束 | agent_id, group_id/session_id, **trigger** | 清除所有状态 |

**trigger 字段**：

| 值 | 来源 | 前端 |
|----|------|------|
| `"user"` | 用户发消息 / DM | 显示"思考中"和"输入中" |
| `"auto"` | 闹钟 / 心跳 | 不在对话界面显示状态 |

## 前端实现

### 状态流转

```
ai_thinking → thinkingAgents.set(agent_id, agent_name)
             → 渲染: MessageBubble thinking={true}（绿色脉冲 + "思考中..."）

ai_typing   → thinkingAgents.delete(agent_id)
            → typingAgents.set(agent_id, agent_name)
            → 渲染: MessageBubble isTyping={true}（"输入中..." + 闪烁光标）

message     → thinkingAgents.delete(sender_id)
            → typingAgents.delete(sender_id)
            → 渲染: 正常消息气泡

ai_thinking_end → thinkingAgents.delete(agent_id)
                → typingAgents.delete(agent_id)  (兜底清理)
```

### MessageBubble 状态 props

| prop | 效果 |
|------|------|
| `thinking={true}` | 绿色脉动光环 + 三点弹跳动画 + "思考中..."标签 + 半透明气泡 |
| `isTyping={true}` | 绿色脉动光环 + 三点弹跳动画 + "输入中..."标签 + 闪烁光标气泡 |
| 两者皆 false | 正常消息渲染 |

### 连发消息展示

AI 在一次回复中多次调用 `send_message` 时：

```
正在思考… → 输入中… → 消息1 → 输入中… → 消息2
```

每条消息之间的间隔由 AI 自然控制（工具调用耗时），用户看到的是"AI 在打字停顿"，而不是一次性吐出一大段。

## 自主行为静默

闹钟和心跳触发的 AI 行为携带 `trigger: "auto"`：

- 对话界面**不显示**"思考中"或"输入中"状态
- AI 发送的消息仍然正常显示在聊天区（因为 message 事件不区分 trigger）
- 用户可在 AI 详情页的"工作区" Tab 看到 AI 当前活动状态

这确保用户不会被 AI 的自主行为打扰——就像你不会注意到邻居是在你敲门之后开门，还是自己主动开门。

## 系统提示词增强

`CORE_IDENTITY` 新增：

```
## 连发消息
你可以在一次回复中连续调用多次 send_message，模拟人类连续表达的习惯——
想到什么就发什么，不需要等思考完整了再一次说完。
思考内容（reasoning）仅用于组织思路，不会发送给他人。
```

AI 被告知：
- 可以连发多条消息（不需要打包成一段）
- reasoning 内容别人看不到（鼓励自由思考）

## 限制与未来增强

| 限制 | 说明 | 未来方向 |
|------|------|---------|
| 消息整段发送 | 当前不逐字打印，消息通过工具调用的原子操作出现 | 如需真正逐字流式，需重构 send_message 为非工具方式 |
| 状态更新无节流 | 快速连发工具时可能频繁切换状态 | 后端去重：同一状态不重复推送 |
| 自主行为状态仅在工作区可见 | AgentDetailPage 尚未接入实时活动状态流 | 后续可加 WebSocket 订阅单个 AI 的活动 |
