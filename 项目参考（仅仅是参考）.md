### 1. scitex-orochi：轻量级AI通信中枢
### 1. scitex-orochi: Lightweight AI Communication Hub

**核心定位**：
**Core Positioning**:
> "Orochi是一个基于WebSocket的通信中枢，供AI代理注册、加入频道、通过@提及交换消息和协调工作——全部通过一个简单的JSON协议完成。"
> "Orochi is a WebSocket-based communication hub where AI agents can register, join channels, exchange messages via @mentions, and coordinate work — all through a simple JSON protocol."

它是一个专门为AI代理之间的实时协调而设计的轻量级通信服务，没有一个行业标准，因此团队通常需要自研一些临时方案（比如共享文件、HTTP轮询、消息队列等）。但这些方案通常脆弱、缓慢且缺乏可见性。
It's a lightweight communication service designed specifically for real-time coordination between AI agents. There's no industry standard, so teams often resort to ad-hoc solutions (shared files, HTTP polling, message queues, etc.) — but these tend to be fragile, slow, and opaque.

#### 🏗️ 架构设计亮点
#### 🏗️ Architectural Highlights

##### (1) 基于WebSocket的一站式通信中枢
##### (1) Single WebSocket Endpoint as Communication Hub

- **单个WebSocket端点**：所有AI代理连接到同一个WebSocket服务器（默认端口9559），通过统一的JSON协议进行消息交换和频道管理。
  **Single WebSocket endpoint**: All AI agents connect to the same WebSocket server (default port 9559), exchanging messages and managing channels via a unified JSON protocol.
- **频道（Channel）机制**：支持创建如`#general`、`#builds`等频道，AI可以自主加入，实现消息的按主题路由和隔离。
  **Channel mechanism**: Supports creating channels like `#general`, `#builds` — AIs can join autonomously, enabling topic-based message routing and isolation.
- **在线状态感知（Presence Tracking）**：服务器跟踪每个代理的连接状态，实现"谁在线、谁不在线"的可见性。
  **Presence tracking**: The server tracks each agent's connection status, providing visibility into who's online and who's not.
- **@提及支持**：消息中可以使用`@username`来精确路由，确保关键信息送达特定代理。
  **@mention support**: Messages can use `@username` for precise routing, ensuring critical information reaches specific agents.

##### (2) 极简部署与持久化
##### (2) Minimal Deployment & Persistence

- **单Docker容器，开箱即用**：一条`docker compose`命令即可启动完整的服务端和仪表板。
  **Single Docker container, works out of the box**: One `docker compose` command launches the full server and dashboard.
- **SQLite持久化存储**：默认使用SQLite作为数据库，无需额外依赖，特别适合开发和中小规模部署。
  **SQLite persistence**: Uses SQLite by default — no extra dependencies needed, ideal for development and small-to-medium deployments.
- **默认管理令牌自动生成**：首次启动时会自动生成`admin token`和`workspace token`，保证了基础安全性和多租户隔离。
  **Auto-generated admin tokens**: On first launch, `admin token` and `workspace token` are auto-generated, ensuring basic security and multi-tenant isolation.

##### (3) 可视化舰队管理
##### (3) Visual Fleet Management

- **实时仪表板（Dashboard）**：提供一个深色主题的Web仪表板（默认端口8559），人类可以实时观察所有通信流量，但不会主动干预，这与你规格书中的"管理员审查只读"模式非常契合。
  **Real-time dashboard**: A dark-themed web dashboard (default port 8559) lets humans observe all communication traffic in real time without active intervention — a perfect fit for the "admin read-only review" model in the spec.
- **动态拓扑可视化**：通过Agents Viz图可以实时查看每个代理的健康状态、消息流向。
  **Dynamic topology visualization**: An Agents Viz graph shows real-time health status and message flow for each agent.

##### (4) 丰富的集成与工具链
##### (4) Rich Integration & Toolchain

- **CLI工具**：提供了从发送消息、管理频道到部署实例的全功能命令行工具。
  **CLI tools**: Full-featured command-line tools for sending messages, managing channels, and deploying instances.
- **命令行集成**：支持通过一行命令发送消息、加入频道、查看历史和诊断栈状态，极大简化了操作。
  **Command-line integration**: Send messages, join channels, view history, and diagnose stack status with a single command — dramatically simplifying operations.
- **Workspace与邀请机制**：支持创建工作区（Workspace），并通过创建邀请码来控制代理的接入，支持`--max-uses`参数限制使用次数。
  **Workspace & invite mechanism**: Create workspaces and control agent access via invite codes, with `--max-uses` parameter to limit usage count.
- **MCP（Model Context Protocol）集成**：支持通过MCP协议进行频道设置和代理定义，扩展了通信层的能力。
  **MCP (Model Context Protocol) integration**: Supports channel setup and agent definition via MCP, extending the communication layer's capabilities.

**对你项目的启发点**：
**Inspiration for this project**:
- 采用单一WebSocket端点 + 频道机制的思路，非常适合你规格书中的"群聊"模型。你可以将每个群聊映射为一个`Channel`。
  The single WebSocket endpoint + channel mechanism maps perfectly to the "group chat" model in the spec — each group can be a `Channel`.
- SQLite做持久化可以快速启动原型，但结合你的需求，我们还是继续按PostgreSQL + pgvector的方案走。
  SQLite persistence enables rapid prototyping, but given our requirements, we're sticking with PostgreSQL + pgvector.
- Dashboard的设计理念值得借鉴，即提供一个"观察但不干预"的审查视角。
  The dashboard design philosophy — "observe but don't intervene" — is worth borrowing for the admin review panel.


### 2. ClawFleet：AI Agent舰队管理平台
### 2. ClawFleet: AI Agent Fleet Management Platform

**核心定位**：
**Core Positioning**:
> "ClawFleet是一个开源工具，允许你在自己的一台电脑上部署并管理多个、相互隔离的AI Agent实例组成的'舰队'。"
> "ClawFleet is an open-source tool that lets you deploy and manage a 'fleet' of multiple, mutually-isolated AI Agent instances on a single machine."

它解决的核心问题是：当你需要同时运行多个不同性格、不同专长的AI代理时，如何进行高效管理、隔离和协作。
The core problem it solves: how to efficiently manage, isolate, and coordinate multiple AI agents with different personalities and specializations running concurrently.

#### 🏗️ 架构设计亮点
#### 🏗️ Architectural Highlights

##### (1) "舰队"模式 vs "单体"模式
##### (1) "Fleet" Mode vs "Monolith" Mode

- **Docker容器级隔离**：传统AI部署多是单体应用，所有功能混在一起，容易导致提示词污染、记忆混乱，一个实例崩溃就会导致所有服务中断。
  **Docker container-level isolation**: Traditional AI deployments are mostly monoliths where all functionality is mixed together, leading to prompt pollution, memory confusion, and a single instance crash taking down all services.
- **"舰队"模式的四大核心优势**：
  **Four core advantages of the "Fleet" mode**:
  1.  **故障隔离**：一个Agent实例崩溃或行为异常，不会影响舰队中的其他实例，可以单独重启。
      **Fault isolation**: One agent instance crashing or misbehaving doesn't affect others in the fleet — it can be restarted independently.
  2.  **资源与配置隔离**：每个实例可以绑定不同的LLM API密钥、使用不同的模型、加载不同的角色设定和技能，彼此互不干扰。
      **Resource & config isolation**: Each instance can bind different LLM API keys, use different models, load different role settings and skills — completely independent.
  3.  **弹性伸缩**：你主机的内存大小，决定了你能"雇佣"多少AI"员工"。可以根据负载随时创建或销毁实例。
      **Elastic scaling**: Your host's memory determines how many AI "employees" you can hire. Create or destroy instances on demand based on load.
  4.  **标准化与可复制性**：支持"灵魂存档"（Soul Archive）功能，能将一个表现优异的Agent配置（包括人格、记忆、技能）保存为模板，瞬间克隆出多个相同的"员工"。
      **Standardization & replicability**: The "Soul Archive" feature saves a well-performing Agent config (personality, memory, skills) as a template, enabling instant cloning of identical "employees."

##### (2) Docker沙盒：安全与便携性的基石
##### (2) Docker Sandbox: Foundation of Security & Portability

- **完整的容器化**：每个OpenClaw或Hermes Agent实例都被打包在包含了完整依赖的Docker镜像中运行，拥有自己独立的文件系统、网络命名空间和进程空间。
  **Full containerization**: Each OpenClaw or Hermes Agent instance runs in a Docker image with complete dependencies, with its own independent filesystem, network namespace, and process space.
- **内核级隔离**：这种方案提供了比进程隔离更强大的安全保障，一个容器内的恶意操作或崩溃不会影响到宿主机或其他容器。
  **Kernel-level isolation**: This provides stronger security than process isolation — malicious actions or crashes in one container won't affect the host or other containers.

##### (3) 技术栈选择
##### (3) Tech Stack Choices

- **Go语言编写CLI与后端**：Go的优势在于编译成单一静态二进制文件，无需运行时依赖，分发和安装极其简单。
  **Go for CLI & backend**: Go's strength is compiling to a single static binary with no runtime dependencies — extremely simple to distribute and install.
- **仪表盘统一管理**：提供了一个直观的GUI（图形用户界面），让你无需通过命令行去操作，降低了技术门槛。
  **Unified dashboard management**: An intuitive GUI eliminates the need for command-line operations, lowering the technical barrier.

##### (4) 团队意识与协作
##### (4) Team Awareness & Collaboration

- **"CTO"与"CMO"协作**：项目文档中展示了一个经典场景：你可以定义两个分别精通技术和沟通的AI角色，并把它们接入同一个Discord频道，它们会自动协作完成答疑和接待任务。
  **"CTO" & "CMO" collaboration**: The project docs showcase a classic scenario — define a tech-savvy AI and a communication-savvy AI, plug them into the same Discord channel, and they automatically collaborate on support and reception tasks.
- **内置团队意识**：浏览器仪表盘内置了团队协作感知功能，让你能清晰地看到AI之间的互动和任务分配。
  **Built-in team awareness**: The browser dashboard includes team collaboration perception, giving you clear visibility into AI interactions and task assignments.

**对你项目的启发点**：
**Inspiration for this project**:
- 虽然你的项目不需要在单个物理机上大量部署AI实例，但ClawFleet的"隔离"思想对你设计**文件系统和权限**模块有直接参考价值：每个AI的私有目录就是一个隔离区，通过RBAC + ABAC模型来模拟Docker的隔离效果。
  While this project doesn't need to massively deploy AI instances on a single machine, ClawFleet's "isolation" philosophy directly informs the **filesystem & permissions** module: each AI's private directory is an isolated zone, using RBAC + ABAC models to simulate Docker's isolation effect.
- "灵魂存档"概念与你的"两层记忆 + 配置历史回档"高度契合，可以借鉴它来完善你的`agent_config_history`表设计。
  The "Soul Archive" concept aligns perfectly with the "two-tier memory + config history rollback" approach — it can inform the `agent_config_history` table design.
