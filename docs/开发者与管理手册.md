## 1. 系统架构

### 1.1 整体架构

```
浏览器/客户端
    │
    ├─ HTTPS ─→ Nginx (443) ─→ Vite 前端 (:5227) ─→ /api 代理 ─→ FastAPI 后端 (:5228)
    │                │
    │                └─ /ws 代理 ─→ FastAPI 后端 (WebSocket)
    │
    └─ 所有请求先到 Nginx，再分发到 Docker 容器
```

### 1.2 Docker 容器

| 容器名 | 内部端口 | 外部端口 | 说明 |
|--------|---------|---------|------|
| `ai_group_postgres` | 5432 | 5432 | PostgreSQL 17 + pgvector |
| `ai_group_backend` | 8000 | 5228 | FastAPI (uvicorn) |
| `ai_group_frontend` | 3000 | 5227 | Vite 开发服务器（含代理） |

### 1.3 前端代理（Vite）

前端 Vite 开发服务器自动代理以下路径到后端，开发环境无需额外配置 Nginx：

| 路径 | 目标 | 类型 |
|------|------|------|
| `/api/*` | `http://backend:8000` | REST（去掉 `/api` 前缀） |
| `/ws` | `ws://backend:8000` | WebSocket |
| `/federation/*` | `http://backend:8000` | REST + WebSocket |

生产环境需要 Nginx 反向代理，见[第 3 节](#3-反向代理配置)。

### 1.4 关键中间件

| 组件 | 位置 | 作用 |
|------|------|------|
| JWT 认证 | `utils/auth.py` | HS256，7 天有效期，Bearer token |
| API Key 加密 | `utils/crypto.py` | Fernet 对称加密存储用户 API Key |
| Embedding 检测 | `utils/embedding.py` | 自动检测 API 提供商并回退 |
| 并发管理 | `services/api_key_concurrency.py` | 纯内存 asyncio.Lock 追踪池 Key 并发 |
| 指标收集 | `services/metrics_collector.py` | p50/p95/p99 延迟统计，60s 批量落库 |

## 2. Docker 部署

### 2.1 环境要求

- Docker Engine ≥ 24 + Docker Compose v2
- 或 Docker Desktop（Windows / Mac / Linux）
- DeepSeek API Key（在 [platform.deepseek.com](https://platform.deepseek.com) 获取）
- 2GB+ 可用内存，5GB+ 可用磁盘

### 2.2 安装与启动

```bash
# 1. 克隆仓库
git clone https://github.com/ShuAICFR/AIsChat.git
cd AIsChat

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env：设置 DB_PASSWORD 和 JWT_SECRET_KEY

# 3. 启动所有服务（首次自动初始化数据库）
docker compose up -d

# 4. 查看状态
docker compose ps
```

### 2.3 日常管理命令

```bash
# 查看日志
docker compose logs -f backend          # 后端日志（实时）
docker compose logs -f frontend         # 前端日志
docker compose logs --tail=100 backend  # 最近 100 行

# 重启服务（代码更新后）
docker compose up -d --build backend    # 仅重建后端
docker compose up -d --build frontend   # 仅重建前端
docker compose up -d --build            # 全部重建

# 进入容器调试
docker compose exec backend bash        # 后端容器
docker compose exec postgres psql -U ai_chat ai_group_chat  # 数据库

# 停止/启动
docker compose down                     # 停止并删除容器
docker compose up -d                    # 重新启动
```

### 2.4 数据目录

| 路径 | 内容 |
|------|------|
| `./data/postgres/` | 数据库文件 |
| `./data/agents/` | AI 文件空间 |
| `./data/groups/` | 群聊文件 |
| `./data/attachments/` | 上传附件 |

通过 `.env` 中的 `DATA_DIR` 环境变量可自定义根目录。

## 3. 反向代理配置

### 3.1 为什么需要反向代理

Docker 容器暴露的是 HTTP 端口（5227/5228），生产环境需要在前面加 Nginx 提供：
- HTTPS/TLS 加密
- 单一入口（80/443 → 内部端口）
- WebSocket 升级支持
- 静态资源缓存

### 3.2 标准 Nginx 配置

```nginx
server {
    listen 443 ssl http2;
    server_name your-domain.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/cert.key;

    # 关键：WebSocket 支持
    location / {
        proxy_pass http://127.0.0.1:5227;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

server {
    listen 80;
    server_name your-domain.com;
    return 301 https://$host$request_uri;
}
```

> **重要**：必须包含 `proxy_set_header Upgrade $http_upgrade` 和 `proxy_set_header Connection "upgrade"`，否则 WebSocket 连接会失败（表现为 "WebSocket is closed before the connection is established"）。

### 3.3 多域名共用证书

如果使用通配符证书（如 `*.example.com`），多个子域名可共用一个 server block：

```nginx
server {
    listen 443 ssl http2;
    server_name app1.example.com app2.example.com api.example.com;
    # ... 其余配置相同
}
```

## 4. FRP 隧道部署

### 4.1 适用场景

当 AIsChat 部署在 NAT 后面的机器（如家庭 NAS），需要通过有公网 IP 的云服务器对外提供服务时，使用 FRP (Fast Reverse Proxy) 建立隧道。

### 4.2 架构

```
用户 → 云服务器 Nginx (443) → frps (8080) → frpc 隧道 → NAS Docker (5227/5228)
```

### 4.3 云服务器 frps 配置

```toml
# /opt/frp/frps.toml
bindPort = 7000
bindAddr = "0.0.0.0"
vhostHTTPPort = 8080

auth.method = "token"
auth.token = "your-frp-token"

webServer.addr = "0.0.0.0"
webServer.port = 7500
webServer.user = "admin"
webServer.password = "dashboard-password"
```

### 4.4 NAS frpc 配置

```toml
# /etc/frp/frpc.toml（在 frpc 容器内）
serverAddr = "your-cloud-server-ip"
serverPort = 7000

auth.method = "token"
auth.token = "your-frp-token"

# 前端
[[proxies]]
name = "aischat-frontend"
type = "http"
localIP = "127.0.0.1"
localPort = 5227
customDomains = ["aischat.your-domain.com"]

# 后端 API
[[proxies]]
name = "aischat-api"
type = "http"
localIP = "127.0.0.1"
localPort = 5228
customDomains = ["aischat-api.your-domain.com"]
```

### 4.5 云服务器 Nginx（配合 FRP）

```nginx
server {
    listen 443 ssl http2;
    server_name aischat.your-domain.com aischat-api.your-domain.com;

    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/cert.key;

    location / {
        proxy_pass http://127.0.0.1:8080;  # frps HTTP 虚拟主机端口
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

> **关键点**：必须为 AIsChat 域名创建**独立**的 nginx server block，不能依赖默认 server。否则 WebSocket 升级请求可能被路由到错误的后端。

## 5. 环境变量参考

### 5.1 必需变量

| 变量 | 说明 | 示例 |
|------|------|------|
| `DB_PASSWORD` | 数据库密码 | `your_secure_password` |
| `JWT_SECRET_KEY` | JWT 签名密钥（同时用于 Fernet 加密） | `至少 32 位随机字符串` |

### 5.2 可选变量

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `DEEPSEEK_BASE_URL` | API 地址（兼容 OpenAI 等） | `https://api.deepseek.com` |
| `DEFAULT_CHAT_MODEL` | 默认聊天模型 | `deepseek-v4-flash` |
| `DEFAULT_WORK_MODEL` | 默认工作模型 | `deepseek-v4-pro` |
| `MODEL_OPTIONS` | 自定义模型列表（JSON） | 按 API 提供商自动设置 |
| `DATA_DIR` | 数据存储根目录 | `./data` |
| `ENCRYPTION_KEY` | API Key 加密密钥 | 复用 `JWT_SECRET_KEY` |
| `AGENT_METRICS_RETENTION_DAYS` | 指标数据保留天数 | `30` |

### 5.3 非 DeepSeek API 配置

```bash
# .env 示例：使用 OpenAI 兼容 API
DEEPSEEK_BASE_URL=https://api.openai.com/v1
DEFAULT_CHAT_MODEL=gpt-4o-mini
DEFAULT_WORK_MODEL=gpt-4o
MODEL_OPTIONS='[{"value":"gpt-4o-mini","label":"GPT-4o Mini"},{"value":"gpt-4o","label":"GPT-4o"}]'
```

系统会自动检测 API 提供商类型，非 DeepSeek API 会自动隐藏不兼容的 `thinking` 参数。

## 7. 联邦通信配置

### 7.1 概念

联邦通信允许两个（或多个）部署在不同服务器上的 AIsChat 实例之间**直接互相转发群聊消息**。这是**服务端之间**的连接——用户浏览器不参与联邦网络。

### 7.2 双层 ID 体系

- **子网 ID**：实例第一次启动时自动生成的 UUID，仅内部使用
- **公网 ID**：`AIsChat-` + 短哈希，由 GitHub 注册表分配，用于跨实例标识

### 7.3 开启步骤

1. **注册公网 ID**：管理员面板 → 联邦 Tab → 编辑实例身份 → 配置 GitHub Token → 一键注册
2. **添加对等端**：输入对方的公网 ID、WebSocket URL（`wss://host/federation/ws`）、共享密钥 → 对方反向添加你
3. **设置群共享**：进入群聊 → 群设置 ⚙️ → 联邦共享 → 勾选对等端
4. **双向配置**：双方**互相**添加对方为对等端，联邦才能正常工作

### 7.4 WebSocket 联邦端点

```
wss://your-domain.com/federation/ws
```

- 通过前端 Vite 代理或 Nginx 直接代理到后端
- 使用 HMAC-SHA256 挑战-应答握手认证
- 共享密钥通过 Fernet 加密存储

### 7.5 URL 动态轮换

管理员可通过三阶段协商协议动态更换对等端地址，防止固定地址长期暴露。详见[联邦 URL 动态轮换协议](./联邦URL动态轮换协议.md)。

### 7.6 注意事项

- 推荐生产环境使用 `wss://`（TLS）而非 `ws://`
- 共享密钥通过安全渠道（加密私信、Signal 等）带外交换
- 目前仅支持群聊联邦，私信（DM）联邦暂未开放
- 两端时钟可能有微小偏差

## 8. 对话日志管理

### 8.1 保留策略

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| 系统硬上限 | 30 条 | 所有 AI 最多保留的对话记录数 |
| 用户默认保留数 | 20 条 | 新用户的默认保留数（≤ 系统上限） |
| 按 AI 覆盖 | 跟随系统 | 管理员可单独调高/调低某个 AI 的保留数 |
| 用户自定义 | 跟随默认 | 用户可调低自己的保留数（≤ 系统上限） |

自动清理：每次保存新对话时，自动删除超出保留上限的最旧记录。

### 8.2 日志内容

每条记录包含：完整 messages 数组（system/assistant/tool）、message_count、token_usage（prompt/completion/total）、模型名称、是否启用深度推理、对话类型（群聊/私信/闹钟）。

### 8.3 管理员三子页签

- **全局设置**：系统硬上限、用户默认保留数、全局用户访问开关
- **按 AI 设置**：选择 AI → 覆盖保留上限（留空继承全局）、用户访问开关
- **查看日志**：选择 AI → 加载列表（含消息数/Token/模型/预览）→ 点击查看完整 JSON

## 9. API Key 池管理

### 9.1 功能

管理员可以添加共享 API Key 到池中，用户即使没有自己的 API Key，只要有兑换的通用额度即可通过池 Key 使用 AI。

### 9.2 Key 属性

| 属性 | 说明 |
|------|------|
| 名称 | 显示名称（如 "DeepSeek Pro"） |
| 优先级 | 数字越大越优先使用 |
| 状态 | 启用/禁用 |
| 并发上限 | 覆盖模型默认值（pro=500, flash=2500），留空=使用默认 |
| 加密 | Key 以 Fernet 加密存储，管理员仅能看到密文后四位 |

### 9.3 四层解析优先链

当 AI 需要调用 LLM 时，按以下优先级获取 API Key：

1. **Agent 自有 Key**：如果创建者给 AI 配置了独立 Key
2. **池绑定 Key**：用户之前已绑定到池中某个 Key（缓存命中）
3. **自动分配**：从池中选负载最低、未冷却的 active Key 并自动绑定
4. **用户自有 Key**：前 3 层都不可用时，使用用户个人 API Key

### 9.4 并发与重试

- 纯内存 `asyncio.Lock` 追踪每个 Key 的并发数
- 429（速率限制）→ Key 冷却 60s，自动换 Key
- 500/503（服务器错误）→ 同 Key 等待 2-3s 重试（最多 2 次）
- 402/401（Key 不可用）→ 通知管理员，跳过换下一个

### 9.5 仪表板

管理员面板 → API 库 → 点击某个 Key 的「详情」→ 查看 Token 消耗趋势图 + 模型分布饼图 + 错误统计。

## 10. 兑换码系统

### 10.1 格式

`RC-` + 16 位十六进制大写字母（如 `RC-A1B2C3D4E5F6G7H8`）。

### 10.2 四种额度类型

| 类型 | 消耗方式 | 说明 |
|------|---------|------|
| `ai_quota` | 每创建 1 个 AI 消耗 1 点 | 创建 AI 的许可 |
| `api_credit` | 1 余额 = 10,000 token | 通用 API 调用额度，pay-as-you-go |
| `agent_bundle` | 创建 AI 时一次性支付 | 该 AI 后续 API 调用全部免费 |
| `file_quota` | 单位 MB | 文件上传空间配额 |

### 10.3 平台赠送额度

管理员可在平台设置中设置 `default_platform_credit`，新用户注册自动获得。修改全局值后通过差值法批量更新所有现有用户。

扣费优先级：先扣 `platform_gifted_credit`（>0 部分），再扣 `api_credit`。

### 10.4 管理员备注

创建兑换码时可添加 `note`（仅管理员可见）、`max_usage`（限制使用次数）、`is_api_pool`（标记为 API 池额度）。

## 11. 备份与恢复

### 11.1 两种备份格式

| | 仅数据库 `.sql` | 完整备份 `.tar.gz` |
|---|---|---|
| 内容 | 表结构 + 数据 | 数据库 + `/app/data/` 全部文件 |
| 大小 | 几十 KB ~ 几 MB | 取决于文件量 |
| 用途 | 快速备份/恢复数据 | 完整迁移到新服务器 |
| 恢复 | `.sql` → 覆盖数据库 | `.tar.gz` → 覆盖数据库 + 文件 |

### 11.2 操作位置

管理面板 → **备份** Tab：
- **完整备份（推荐）**：一键下载 `.tar.gz`
- **仅数据库**：快速导出 `.sql`
- **恢复**：上传对应文件即可恢复

> ⚠️ 恢复操作会覆盖当前数据。仅数据库备份**不含**附件和 AI 文件，迁移务必使用完整备份。

## 12. 性能调优

### 12.1 数据库

```bash
# 进入数据库容器
docker compose exec postgres psql -U ai_chat ai_group_chat

# 手动 VACUUM（清理死元组）
VACUUM ANALYZE;

# 训练 ivfflat 索引（向量搜索加速，需有一定数据量后执行）
-- 在 psql 中执行：
-- ALTER INDEX rough_memories_embedding_idx SET (lists = 100);
```

### 12.2 Docker 资源

- 建议给 Docker 分配 ≥ 4GB 内存
- 如果机器内存紧张，可在 docker-compose.yml 中限制容器内存

### 12.3 清理旧数据

## 15. 代码架构

### 15.1 后端分层

```
routers/   → 薄层，参数校验和 HTTP 状态码转换
services/  → 业务逻辑，操作 ORM 模型，返回 ORM 对象或 dict
models/    → SQLAlchemy 2.0 ORM
schemas/   → Pydantic v2 请求/响应模型
utils/     → 横切工具：JWT、密码哈希、API Key 加密、Embedding 检测
```

### 15.2 关键约定

- 所有数据库操作使用 `async/await`，通过 `get_db` 依赖注入获取 `AsyncSession`
- `get_db` 在请求成功时自动 commit，异常时自动 rollback
- 业务异常用 `raise ValueError("中文消息")`，router 层捕获后转 HTTPException
- ORM → dict 转换由 service 层的 `*_to_dict()` 函数完成

### 15.3 数据库注意事项

- 使用 `pgvector/pgvector:pg17` 镜像（v0.6.0 起从 pg16 升级）
- 12 张表由 `init-db.sql` 创建，无需 Alembic 迁移（当前阶段）
- 表名是 `groups`（不是 `group`，避免 SQL 关键字冲突）

## 16. 常见部署问题

### 16.1 启动失败

**Q: `docker compose up -d` 失败？**
- 确认安装了 Docker Desktop 或 Docker Engine + Compose v2
- 检查 `.env` 文件存在且 `DB_PASSWORD`、`JWT_SECRET_KEY` 已设置
- 检查端口 5227、5228、5432 未被占用：`netstat -tlnp | grep -E "5227|5228|5432"`

**Q: 数据库连接失败？**
- 检查 PostgreSQL 容器是否健康：`docker compose ps`
- 首次启动等 10 秒让 PostgreSQL 初始化完成
- 检查 `DB_PASSWORD` 与 `init-db.sql` 中的密码一致

**Q: 后端启动失败？**
```bash
docker compose logs backend | grep -i error
# 常见原因：.env 中的特殊字符未正确转义、API Key 配置错误
```

### 16.2 AI 不回复

1. 检查 AI 状态：是否被设为 `offline` 或 `blocked`
2. 检查 API Key 是否有效且有余额
3. 查看后端日志：`docker compose logs -f backend | grep -E "ERROR|429|401|402"`
4. 如果不能使用自己的 Key，检查管理员是否配置了 API Key 池

### 16.3 性能下降

- 数据库膨胀：运行 `docker compose exec postgres vacuumdb -U ai_chat ai_group_chat`
- 磁盘不足：清理 `data/` 目录中不需要的文件
- 内存不足：增加 Docker 内存限制或关闭不用的容器

## 17. WebSocket 问题专项

### 17.1 问题诊断流程

WebSocket 连接失败（"WebSocket is closed before the connection is established"）的排查步骤：

1. **确认容器运行**：`docker compose ps | grep ai_group`（三个容器应全部 Up）
2. **确认端口监听**：`ss -tlnp | grep -E "5227|5228"`
3. **检查 Nginx 是否有对应 server block**：`grep -r "your-domain" /etc/nginx/`
4. **检查 Nginx 是否有 WebSocket 头**：
   ```nginx
   proxy_set_header Upgrade $http_upgrade;
   proxy_set_header Connection "upgrade";
   ```
5. **从服务器内部测试**：
   ```bash
   # 测试 WebSocket 升级是否通过代理链
   curl -i -N -H "Connection: Upgrade" -H "Upgrade: websocket" \
     -H "Host: your-domain.com" \
     -H "Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==" \
     -H "Sec-WebSocket-Version: 13" \
     http://127.0.0.1:8080/ws?token=test
   # 预期返回 HTTP 403（后端拒绝了无效 token —— 说明链路通）
   ```
6. **从外网测试**（需要 Python websockets 库）：
   ```python
   import asyncio, websockets, ssl
   async def test():
       ssl_ctx = ssl.create_default_context()
       try:
           async with websockets.connect('wss://your-domain.com/ws?token=test', ssl=ssl_ctx):
               print('OK')
       except websockets.exceptions.InvalidStatusCode as e:
           print(f'Backend responded: HTTP {e.status_code}')  # 403 = good, 502 = bad
   asyncio.run(test())
   ```

### 17.2 常见原因速查

| 症状 | 可能原因 | 解决 |
|------|---------|------|
| `connection refused` | 前端容器未运行 | `docker compose up -d` |
| `closed before established` | Nginx 缺少 Upgrade 头 | 添加 `proxy_set_header Upgrade/Connection` |
| `closed before established` | Nginx 路由到错误的 server | 检查 server_name 是否匹配，是否需独立 server block |
| `HTTP 502` | 后端未运行或 FRP 隧道断开 | 检查 Docker 和 FRP 状态 |
| `HTTP 403` | 正常（无效 token），用真 token 即可 | — |
| `TIMEOUT` | 防火墙/安全组未开放端口 | 检查 443 端口 |

### 17.3 FRP 环境特别注意

在 FRP 隧道部署中，WebSocket 经过多跳链路。确保：
1. **云服务器 Nginx 有该域名的独立 server block**（不能依赖默认 server）
2. **Nginx server block 代理到 frps 的 HTTP 虚拟主机端口**（默认 8080）
3. **FRP 配置中 `type = "http"`**，HTTP 代理模式天然支持 WebSocket
4. **SSL 证书覆盖该子域名**（推荐通配符证书）

## 18. 联邦通信排错

### 18.1 连接失败

1. 确认对方公网 URL 从你的服务器可直接访问
2. 确认共享密钥双方一致
3. 确认对方公网 ID 与你的对等端配置一致
4. 如果使用 `wss://`，确认 TLS 证书有效
5. 如果通过 Nginx 反代，确认配置了 WebSocket Upgrade
6. 查看后端日志：`docker compose logs -f backend | grep "🌐"`

### 18.2 消息收不到

1. 确认双方都已开启群聊的"联邦共享"开关
2. 确认对等端状态为绿色"已连接"
3. 检查共享方向（bidirectional / incoming / outgoing）
4. 两端的 `federation_peers` 表数据是否一致

### 18.3 URL 轮换失败

1. 确认新 URL 格式正确（`wss://host:port/federation/ws`）
2. 确认新 URL 可从发起方直接访问
3. 两次轮换必须间隔 ≥ 5 分钟
4. 查看后端日志中的轮换状态
